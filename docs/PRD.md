# AI 建站平台 — 产品需求文档（PRD）

**版本**：v1.0  
**更新日期**：2025-02-20  
**角色**：产品经理 / UIUX 设计师

---

## 一、产品概述

### 1.1 产品愿景

打造一款**通过自然语言对话即可创建、发布、管理静态网站**的 AI 建站平台。用户无需写代码或学习建站工具，只需在聊天框中描述需求，平台基于 DeepSeek API 生成**采用完整 Tailwind CSS 的静态网站**并自动发布到 GitHub Pages，提供访问链接与源码下载，支持后续手动编辑内容。

### 1.2 目标用户

| 用户类型 | 典型场景 | 核心诉求 |
|---------|----------|----------|
| 个人/创作者 | 个人主页、作品集、博客 | 零代码、快速上线、可自定义 |
| 小微团队/创业 | 落地页、产品介绍、活动页 | 低成本、快速验证、可编辑 |
| 教育/学生 | 课程作业、项目展示 | 简单、免费、可下载源码学习 |

### 1.3 核心价值主张

- **对话即建站**：用自然语言描述，AI 理解并生成完整静态站。
- **一键发布**：自动推送到 GitHub Pages，获得可访问的 URL。
- **可掌控**：支持在线编辑内容、下载完整压缩包，数据可迁移。

## 二、用户故事与场景

### 2.1 主流程用户故事

- **US-01**：作为用户，我可以在首页输入“帮我做一个个人作品集，包含关于我、项目、联系方式”，以便 AI 帮我生成对应静态网站。
- **US-02**：作为用户，我可以在生成完成后获得一个可点击的访问链接，以便在浏览器中查看网站。
- **US-03**：作为用户，我可以将网站自动发布到 GitHub Pages，以便获得稳定、免费的托管与自定义域名能力（可选）。
- **US-04**：作为用户，我可以下载完整静态网站压缩包，以便本地备份或迁移到其他托管。
- **US-05**：作为用户，我可以在平台内手动编辑网站的文字、图片等内容，以便微调而不必重新生成。

### 2.2 关键场景

1. **首次建站**：打开平台 → 输入需求 → 等待生成 → 查看预览 → 发布/下载/编辑。
2. **内容修改**：网站一旦生成后，不支持通过 AI 再次调整；需进入「编辑内容」使用**可视化区块编辑**修改文案、图片等，保存后刷新预览/重新发布。

## 三、功能需求

### 3.1 功能清单（MoSCoW）

#### Must Have（必须有）

| ID | 功能 | 描述 | 验收标准 |
|----|------|------|----------|
| F-01 | 首页聊天建站 | 首页以聊天输入框为主，用户输入建站需求 | 单次描述生成站点，生成后不可再通过 AI 修改 |
| F-02 | AI 生成静态站 | 调用 DeepSeek API，根据对话生成**完整 Tailwind CSS 源码**（HTML + Tailwind 类 + 必要 JS） | 生成可独立运行的静态站点，样式全部基于 Tailwind，支持多页 |
| F-03 | 实时/生成后预览 | 生成完成后在平台内可预览网站 | 新开标签或 iframe 内预览，样式与最终一致 |
| F-04 | 发布到 GitHub Pages | 将生成站点推送到用户或平台 GitHub 仓库并开启 Pages | 用户获得 `*.github.io` 或自定义域名访问链接 |
| F-05 | 访问链接展示 | 展示可点击的访问域名（如 `https://xxx.github.io/yyy`） | 点击即可在浏览器打开已发布站点 |
| F-06 | 下载静态站压缩包 | 提供「下载」按钮，打包当前站点为 zip | 解压后为**完整 Tailwind CSS 静态源码**，可本地打开或自托管 |
| F-07 | 可视化编辑内容 | 用户以区块形式编辑页面标题、段落、图片等，不编辑源代码 | 编辑后能保存、刷新预览，可选重新发布 |

#### Should Have（应该有）

| ID | 功能 | 描述 |
|----|------|------|
| F-08 | 对话历史与项目列表 | 保存每次建站对话与对应项目，可回到历史项目继续编辑/发布 |
| F-09 | （已移除）多轮对话优化站点 | 网站生成后仅支持可视化编辑，不支持 AI 再次修改 |
| F-10 | 发布状态与日志 | 展示发布进度（创建仓库、推送、Pages 开启），失败时给出原因与重试 |

#### Could Have（可以有）

| ID | 功能 | 描述 |
|----|------|------|
| F-11 | 用户账号体系 | 注册/登录，将项目与用户绑定 |
| F-12 | 自定义域名绑定说明 | 文档或引导用户如何在 GitHub Pages 绑定自己的域名 |
| F-13 | 模板/示例 | 提供“个人主页”“落地页”等示例描述，一键填充到聊天 |

### 3.2 功能详细说明

**F-01 首页聊天建站**

- 首屏以**单一主输入框**为核心，辅以简短说明（如“描述你想做的网站”）。
- **单次生成**：用户描述需求后 AI 生成站点；生成完成后不支持通过对话再次修改，如需调整请使用「编辑内容」进行可视化编辑。
- 输入限制：单条建议 2000 字以内，避免 token 超限；可支持粘贴长文本。

**F-02 AI 生成静态站**

- 后端将对话历史整理为 prompt，调用 DeepSeek API（流式或非流式视实现选择）。
- **生成规范**：生成的静态网站必须为**完整 Tailwind CSS 代码**——HTML 使用 Tailwind 工具类（utility classes）进行布局与样式，不依赖手写大段自定义 CSS；页面中通过 Tailwind CDN 或内置构建引入 Tailwind，确保下载后的源码可独立运行且风格统一、易维护。
- AI 输出需为**结构化内容**：如约定 JSON 描述页面结构 + 各文件内容；由后端解析并落盘为静态资源。至少支持：多 HTML 页面、Tailwind 类驱动的样式、必要 JS（如导航、简单交互）。

**F-04 发布到 GitHub Pages**

- **方式 A**：用户授权 GitHub OAuth，平台在其账号下创建仓库并推送，由用户或平台开启 Pages。
- **方式 B**：平台使用组织账号创建仓库，每个项目对应一仓库或一分支，生成固定 URL 规则（如 `https://platform.github.io/project-id`）。
- 需考虑：token 安全存储、仓库命名唯一性、Pages 默认从 main 分支的 /root 或 /docs 提供。

**F-07 可视化编辑内容**

- 编辑入口：从「项目详情/预览」页进入「编辑内容」。
- 编辑方式：**仅支持可视化区块编辑**——将页面拆为可编辑区块（标题、段落、图片），在线表单编辑文案与图片地址等，写回 HTML；不提供网页源代码编辑。
- 保存后：更新预览、更新可下载的 zip、可选“重新发布”到 GitHub。

## 四、非功能需求

| 类型 | 要求 |
|------|------|
| 性能 | 首屏可交互 &lt; 3s；AI 首 token 响应建议 &lt; 5s（依赖 DeepSeek）；生成完整站点 &lt; 60s（视页面数量） |
| 可用性 | 核心流程无需注册即可体验（可先以匿名项目 + 本地存储）；发布与下载需明确引导 |
| 安全 | 用户 GitHub Token 仅用于仓库操作，加密存储或短期使用；不记录敏感输入 |
| 兼容 | 生成站点适配现代浏览器（Chrome/Firefox/Safari/Edge 近两版）；管理端建议 Chrome/Edge |
| 可维护 | 日志记录生成请求、发布结果；便于排查 DeepSeek 与 GitHub API 问题 |

## 五、UI/UX 设计

### 5.1 设计原则

- **对话优先**：首屏减少干扰，突出“输入需求 → 得到网站”的心智。
- **进度可见**：生成与发布过程有明确状态（思考中 → 生成中 → 预览就绪 → 发布中 → 完成）。
- **结果可操作**：预览、链接、下载、编辑入口集中且清晰，减少二次学习成本。

### 5.2 信息架构（IA）

```
首页（建站入口）
  ├── 聊天区（输入 + 对话历史）
  ├── 当前项目状态（未生成 / 生成中 / 可预览 / 已发布）
  └── 快捷操作（预览 | 发布 | 下载 | 编辑）

项目/会话列表（Should Have）
  └── 每个项目 → 详情（同首页当前项目能力）

项目详情 / 预览
  ├── 预览 iframe / 新开页
  ├── 操作栏：访问链接 | 下载 | 编辑
  └── 编辑入口 → 编辑模式（区块编辑 or 源码编辑）
```

### 5.3 页面与交互设计

#### 5.3.1 首页（建站首页）

- **布局**：  
  - 顶部：Logo + 产品名（可选：项目列表入口、设置/帮助）。  
  - 中部：**主聊天输入框**（多行，占视觉重心），占位符如“描述你想做的网站，例如：做一个个人作品集，包含关于我、项目展示和联系方式”。  
  - 下方或侧边：当前会话的**对话气泡列表**（用户消息 / AI 回复），AI 回复中可包含“正在生成站点…”占位与进度。  
  - 生成完成后：在对话下方或侧边栏出现**卡片**：预览缩略图、站点标题、**访问链接**、**下载**、**编辑** 按钮。

- **交互**：  
  - 输入框支持 Enter 发送，Shift+Enter 换行。  
  - 发送后输入框禁用或显示“AI 正在思考…”，避免重复提交。  
  - 生成过程中可展示步骤：解析需求 → 生成结构 → 生成页面 → 就绪。  
  - 链接为可点击的 `https://...`，下载为一次点击即开始下载 zip。

- **空状态**：无对话时，输入框上方可有一句引导语 + 2–3 个示例短句（如“个人作品集”“产品落地页”“活动报名页”）点击即填入输入框。

#### 5.3.2 预览与结果区

- **预览**：  
  - 方式一：当前页内 iframe，右侧或下方为预览区，随生成/编辑实时刷新。  
  - 方式二：独立「预览」按钮新开标签，URL 为平台提供的预览地址（需后端支持预览路由）。  

- **结果区内容**：  
  - 标题：如“你的网站已就绪”。  
  - 主 CTA：**打开网站**（跳转至 GitHub Pages 或预览 URL）。  
  - 次要：**下载压缩包**、**编辑内容**。  
  - 若已发布：显示完整 URL 与“已复制”的复制按钮。

#### 5.3.3 编辑模式

- **入口**：从首页结果卡或项目详情点击「编辑内容」。
- **布局**：  
  - 左侧或上半：**区块编辑区**（按页面切换 HTML 文件后，列出可编辑块：标题、段落、图片，每块对应输入框/图片 URL）。  
  - 右侧或下半：**实时预览**（iframe），保存后刷新。  
- **区块编辑**：  
  - 列出可编辑块（如“主标题”“自我介绍”“项目 1 标题/描述/图片”），每块对应输入框或图片地址；保存后写回 HTML 并刷新预览。  
- **顶部操作**：保存、返回项目、重新发布（可选）。

### 5.4 视觉与规范建议

- **风格**：简洁、偏工具/生产力风格；避免过于“卡通”，以建立专业感。  
- **色彩**：  
  - 主色：一种主色（如蓝或绿）用于主按钮、链接、关键状态。  
  - 背景：浅色背景（#f8f9fa 或 #fff），深色文字；可选深色模式。  
  - 状态色：成功绿、进行中蓝、错误红、等待灰。  
- **字体**：  
  - 界面：无衬线（如 Inter、PingFang SC、System UI）。  
  - 代码编辑：等宽字体（Monaco、Fira Code）。  
- **间距与圆角**：统一 4/8/16/24px 间距体系；输入框与卡片圆角 8px，按钮 6–8px。  
- **响应式**：桌面优先；移动端保证输入、发送、查看链接与下载可用，可考虑对话与预览上下堆叠。

### 5.5 关键交互状态

| 状态 | 表现 |
|------|------|
| 输入中 | 输入框可编辑，发送按钮可用 |
| 已发送等待 AI | 输入框禁用或占位“正在思考…”，可显示 loading 动画 |
| 生成中 | 步骤条或进度文案；“生成完成”后结果区出现 |
| 发布中 | “发布到 GitHub” 按钮变为“发布中…”，禁止重复点击 |
| 发布成功/失败 | Toast 或行内提示；成功展示链接与复制按钮 |
| 编辑中 | 编辑区与预览区联动，保存后预览刷新 |

## 六、成功指标与验收

- **功能验收**：完成 F-01～F-07 全部主流程，且无阻断性 bug。  
- **体验验收**：新用户从打开首页到获得可访问链接，在 5 分钟内可完成（含一次生成 + 发布）。  
- **可选指标**：生成成功率 &gt; 95%（可解析、可预览）；发布成功率 &gt; 90%（在 GitHub 配置正确前提下）。

## 七、附录：名词与缩写

| 术语 | 说明 |
|------|------|
| 静态网站 | 仅含 HTML/CSS/JS 等静态资源，无需服务端渲染 |
| Tailwind CSS | 本平台生成的站点统一使用 Tailwind 工具类编写样式，输出为完整 Tailwind 源码，可独立运行 |
| GitHub Pages | GitHub 提供的静态站点托管服务，支持 `*.github.io` 域名 |
| DeepSeek API | 用于理解用户需求并生成页面结构与内容的 AI 接口 |
| 发布 | 将生成后的静态文件推送到 GitHub 仓库并开启 Pages |
